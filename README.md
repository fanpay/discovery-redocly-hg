# Redocly MCP Integration — Tutorial Project

This project demonstrates the full MCP (Model Context Protocol) integration available in Redocly Realm,
using the Museum API as a real-world example. It covers all five pieces of the integration end to end.

---

## What is MCP in Redocly?

Model Context Protocol (MCP) is a standard that lets AI coding assistants (Cursor, VS Code Copilot,
Claude Code, etc.) connect to an external server and receive structured, live context — in this case,
your API documentation and the ability to call your API.

Redocly Realm generates an MCP server automatically at `/mcp` on your project URL. You don't build it;
you configure what it exposes.

---

## The Five Pieces

### 1. MCP Server (auto-generated by Realm)

**Where:** No file — it's a Realm feature enabled by default.

**What it does:** Exposes your docs and API operations as MCP tools at `https://your-project.com/mcp`.
AI clients connect here to get live context about your API.

**Why it matters:** This is the backbone. Without it, nothing else works. All the other pieces either
configure it, document it, or connect to it.

---

### 2. `mcp:` config in `redocly.yaml`

**File:** [`redocly.yaml`](redocly.yaml)

```yaml
mcp:
  hide: false
  docs:
    hide: false
    name: "Museum API Docs"
    ignore:
      - style-guide.md
```

**What it does:**
- `hide: false` — keeps MCP enabled globally (set to `true` to disable everything)
- `docs.name` — the display name MCP clients show during connection setup
- `docs.ignore` — pages excluded from the MCP server context. `style-guide.md` is internal
  tooling content that has no value for AI assistants helping API consumers.

**Why this approach:** Keeping the MCP context focused on developer-facing content (API reference,
guides) and excluding internal/editorial pages improves AI answer quality. Feeding irrelevant
content to an LLM just adds noise.

---

### 3. `x-mcp` extension in `apis/openapi.yaml`

**File:** [`apis/openapi.yaml`](apis/openapi.yaml)

```yaml
x-mcp:
  protocolVersion: "2024-11-05"
  capabilities:
    tools: {}
  tools:
    - name: getMuseumHours
      description: Get upcoming museum operating hours for a given date range.
    - name: listSpecialEvents
      description: List all upcoming special events at the museum.
    - name: getSpecialEvent
      description: Get full details about a specific special event by its ID.
    - name: buyMuseumTickets
      description: Purchase general admission or special event tickets for the museum.
    - name: getTicketCode
      description: Retrieve the QR code image for a purchased ticket.
```

**What it does:** Declares which API operations are available as MCP tools, and what description
the AI sees for each one. The description is critical — it's how the AI decides when and whether
to call each tool.

**Why these tools and not all of them:**
- `createSpecialEvent`, `updateSpecialEvent`, `deleteSpecialEvent` are **intentionally excluded**.
  These are admin-only mutations that could have irreversible side effects if an AI calls them
  without deliberate human intent. Never expose destructive or privileged operations via MCP
  unless you have explicit auth gating in place.
- The included tools are read-only or safe user-facing actions (buying a ticket is user-initiated
  by definition).

---

### 4. `{% connect-mcp /%}` Markdoc tag

**File:** [`mcp-connect.md`](mcp-connect.md)

```markdoc
{% connect-mcp placement="bottom" alignment="start" options=["cursor", "vscode", "copy"] /%}
```

**What it does:** Renders a UI button on the page with a dropdown. Visitors click it to connect
their editor to the `/mcp` server with one click — no manual configuration needed on their part.

**Options used:**
- `placement="bottom"` — dropdown opens downward (fits mid-page layouts)
- `alignment="start"` — aligns to the left edge of the button
- `options=["cursor", "vscode", "copy"]` — shows all three connection methods

**Why a dedicated page:** Putting the connect button on its own `mcp-connect.md` page (rather than
inline in the index or API reference) makes it discoverable without cluttering other pages. It also
gives space to explain what the user is connecting to and which tools are available.

---

### 5. Navigation entry in `sidebars.yaml`

**File:** [`sidebars.yaml`](sidebars.yaml)

```yaml
- page: mcp-connect.md
  label: Connect with AI
```

**What it does:** Makes the page visible in the sidebar under the label "Connect with AI".

**Why this label:** "Connect with AI" is user-goal language — it says what you'll accomplish,
not what the technology is called. Visitors scanning the sidebar understand immediately what
the page is for.

## How the Five Pieces Interact

```
redocly.yaml (mcp: config)
        │
        ▼
Realm generates /mcp endpoint  ◄── openapi.yaml (x-mcp: tools definition)
        │
        ▼
User lands on mcp-connect.md
        │
        ▼
{% connect-mcp /%} button → user connects Cursor/VS Code/Claude Code
        │
        ▼
AI assistant calls /mcp → gets live context → helps user write code
```

## Recommendations

### Security
- **Never expose admin or destructive operations via `x-mcp`** unless you have authentication
  in place at the MCP server level. The `mcp.docs.ignore` option helps with docs content, but
  `x-mcp` is your gate for API tool exposure.
- If your Realm project requires authentication, configure the `Authorization` header in the
  MCP client config. The MCP server supports custom headers.

### Content quality
- Write good `description` strings in `x-mcp.tools`. The AI uses them to decide which tool to
  call. Vague descriptions lead to the wrong tool being called.
- Use `mcp.docs.ignore` to exclude pages that aren't useful for API consumers (changelogs,
  internal style guides, migration guides that are no longer relevant).

### UX
- Place `{% connect-mcp /%}` near the top of your quickstart or getting-started page — the moment
  a developer decides to try your API is when they want to connect their tools.
- Use `options=["cursor", "vscode", "copy"]` to support all tool ecosystems. Don't pre-filter
  unless you know your audience uses only one tool.

### Iteration
- Start with a small `x-mcp` tool set (read-only ops). Add write operations only after you've
  tested the AI behavior and are confident the descriptions are precise enough to prevent
  unintended calls.
- Monitor which tools the AI invokes most via your API logs — that tells you what developers
  actually need help with.
